# Worktree Memory Strategy

> A worktree is a clone of the main repo, not an independent project. There is only one copy of memory.

## Activation Condition

Automatically activates when the `~/.claude/projects/` project key contains `-worktrees-`.

## Parent Project Resolution

Worktree project key pattern: `{parent}-worktrees-{branch}`

```
D--my-project-worktrees-feat-my-feature-PROJ-26
              ↓ take everything before -worktrees-
D--my-project  ← parent project key
```

Resolution method: take everything before `-worktrees-` as the parent project key.

**Validation**: the resolved parent key must correspond to an existing directory under `~/.claude/projects/`. If the directory does not exist, fall back to glob search (same as session-management.md step 3). Never create a path from thin air.

## Memory Reading

**Always read from the parent.**

```
~/.claude/projects/{parent}/memory/MEMORY.md     ← Hot memory
~/.claude/projects/{parent}/memory/*.md           ← Warm memory
```

Do not read memory from the worktree's own project directory (it should be empty).

## Memory Writing

**Always write to the parent.**

- New feedback / reference / project memory → write to `{parent}/memory/`
- Do not create a `memory/` directory in the worktree's project directory
- Feature-specific memory uses the naming convention `wt-{ticket}-{topic}.md` (going forward; legacy files like `wt-docker-cicd.md` without a ticket ID do not need to be renamed)
- When a feature is complete, `wt-` prefixed memory files should be archived (add `archived: true`)

## Session Writing

**Write to the parent's sessions directory, tagged with the branch.**

```
Path: ~/.claude/projects/{parent}/sessions/{date}.md
```

Tag each entry with the branch for multi-worktree progress tracking:

```markdown
## Completed
- ✅ abc1234 feat: add CSV upload (PROJ-26) [branch: feat/my-feature-PROJ-26]
- ✅ def5678 fix: header alignment (PROJ-85) [branch: fix/my-fix-PROJ-85]
```

Entries from different worktrees on the same day are appended to the same note (consistent with the existing "append, don't overwrite" rule).

**Note**: Claude Code's file writes are not atomic appends. If two worktree sessions commit simultaneously, session notes may overwrite each other. In practice this rarely happens (human-paced operations), but if it does, verify with `git diff` and fix manually.

## Anti-Trampling Rules

When multiple worktrees are being developed in parallel, avoid interfering with each other:

| Allowed | Not Allowed |
|---------|------------|
| Read all of parent's memory | Modify another branch's session entries |
| Append your own branch's session entries | Delete or overwrite parent's memory |
| Create new `wt-{your-ticket}` prefixed memory | Modify `wt-{other-ticket}` memory |
| Read parent's sessions (to understand overall progress) | Write worktree-specific decisions into general memory |

## Subagent Worktree Mode

When the main session spawns a worktree agent, the prompt must inject:

```
Parent project: ~/.claude/projects/{parent}/
Memory path: ~/.claude/projects/{parent}/memory/
Current branch: {branch-name}
```

This ensures the agent knows where to read memory, rather than starting from an empty worktree project directory.

## Existing Worktree Directory Cleanup

Existing empty worktree project directories (no memory/, no sessions/) can be safely ignored.
Directories containing only session history (`.jsonl`) are auto-generated by Claude Code and do not affect this strategy.
